# Alias Normalization Hard NIL 优化建议

## 1. 当前问题

专项集的 20 条 hard NIL 已确认不属于运行 KB 的规范名或 alias。候选级 Negative Precision 为 75%（15/20）：`中国能源研究会`、`中国农业银行`、`北京协和医院`、`上交`、`深圳大学` 会被 fuzzy containment 规则带入候选集。

根因不是 alias 精确索引错误，而是本地 fallback 的 fuzzy 规则将任意单向子串包含视为候选证据，并固定赋予 0.85 分；该阶段不使用实体类型或上下文。完整 pipeline 默认 NIL 阈值为 0.90，理论上能拒绝 0.85 fuzzy 候选，但该保护依赖运行配置，且候选污染仍会影响阈值调整、后续排序与 trace 可解释性。

## 2. Badcase 分析摘要

|输入|误召回实体|直接原因|应拒绝原因|
|-|-|-|-|
|中国能源研究会|国家能源投资集团有限责任公司|短 alias `国能` 被包含|研究会与能源集团不是同一实体|
|中国农业银行|中国农业银行股份有限公司|输入是全称前缀/覆盖 `农业银行`|严格 KB alias 词表未登记该形式；也是应评审的 alias 覆盖边界|
|北京协和医院|北京市|地区名 `北京` 被包含|医疗机构不能退化为地区实体|
|上交|上海证券交易所|`上交`是`上交所`前缀|高校语境与证券交易所不符|
|深圳大学|深圳市|地区名 `深圳` 被包含|高校不是地区实体|

完整逐例证据见 [alias_nil_badcase_analysis.md](../reports/alias_nil_badcase_analysis.md)。

## 3. 优化方案

### 方案 A：低风险——收紧 fuzzy containment 规则（推荐先做）

**建议修改文件：** `entity_linker/pipeline.py` 的 `_LocalKnowledgeBase.get_entities_by_alias_fuzzy()` 与 `_FallbackCandidateGenerator.generate()`；同步补充 alias 专项回归断言。

**建议规则：**

- 精确 alias 路径不变；
- 对单向包含的 fuzzy 命中增加最小 alias 长度和长度比例门槛，例如两字符及以下 alias 不参与 substring fuzzy；
- 对 `mention` 仅包含地区、简称等短 alias 的情况拒绝生成 fuzzy 候选；
- 将 fuzzy 证据改为可解释分数（长度比例/编辑距离），而不是固定 0.85；
- 保持原 NIL 阈值语义，避免用“降低阈值”掩盖错误。

**影响范围：** 仅本地 fallback 候选生成；精确 alias 正例不受影响，可能减少部分依赖子串模糊命中的召回。

**预期：** 本批 5 条中地区/短简称驱动的 4 条应被直接拒绝；候选级 Negative Precision 预计从 75% 提升到约 90% 以上。`中国农业银行`属于合理别名覆盖边界，不应承诺由该规则强制拒绝。

**风险：** 历史数据中依赖短模糊 alias 的真实正例可能转为 NIL，需以原 120 条正例与主链接集回归评估，而不是只看 hard NIL。

### 方案 B：中风险——增加实体类型与上下文约束

**建议修改文件：** `entity_linker/pipeline.py`（候选生成接口接收 `mention_type`/context、调用点传入已识别 mention 信息），可能涉及 `entity_linker/adapters.py` 或模型元数据保留，以获取 KB 的摘要/业务/关键词。

**建议机制：**

- 候选生成或排序阶段使用已识别 mention 的类型过滤：医疗/教育/机构类输入不得因地名子串只保留 REGION 候选；
- 使用上下文关键词与实体行业、summary、keywords 做轻量重合度加分；
- 当候选类型与 mention 类型强冲突且不存在精确 alias 时，直接进入 NIL 或降低至低于 NIL 阈值。

**影响范围：** 本地候选与消歧接口契约、测试和 trace evidence；属于第三成员实体链接模块的增量增强，但需要与总体 pipeline 接口保持兼容。

**预期：** 可针对北京协和医院、上交、深圳大学等上下文/类型明确的负例提高拒绝率，并减少“地区/证券候选抢占”的问题；候选级 Negative Precision 预计可达到 90% 左右。

**风险：** NER/上游提供的类型可能为 UNKNOWN 或不准确，过强过滤会伤害跨类型别名和多义实体；KB 当前 `summary` 等字段在运行适配器中的保留程度需要先审计。

### 方案 C：高风险——候选重排与 LLM 兜底

**建议修改文件：** `entity_linker/pipeline.py` 的候选排序/`_disambiguate_mention()`，以及既有 BGE、LLM 抽象层与接口开关；需补充端到端评测脚本和 trace schema 的候选分数、触发原因记录。

**建议机制：**

- 对 fuzzy 召回候选使用 BGE/重排器计算 mention+context 与实体描述的语义相关度；
- 小 margin、候选类型冲突或低分时触发 LLM 兜底，并要求输出结构化拒绝/选择证据；
- 使用独立验证集校准 NIL 阈值，而不是只针对 5 条调参。

**影响范围：** 候选排序、模型资源、服务时延、外部依赖和 trace；超出纯 alias lookup 的轻量改动。

**预期：** 在复杂上下文、母子公司和同名歧义上可明显提升真实最终链接质量；对 hard NIL 的最终拒绝更稳健，但候选级“无候选”指标不一定变为 100%。

**风险：** 模型加载与推理成本、结果波动、LLM 可复现性及评测泄漏风险更高；不建议仅为消除当前 5 条候选误召回而优先实施。

## 4. 推荐路径与是否值得修改

建议先实施 **方案 A**，但先增加两类回归：

1. 对 160 条 alias 专项集同时报告候选级 Negative Precision 与完整 pipeline 的最终 NIL 指标；
2. 对原 120 条正向 alias 和主链接集分别检查召回变化。

若方案 A 明显降低真实简称召回，或默认/实际接口阈值下仍出现最终错误链接，再进入方案 B。方案 C 应作为后续模型化消歧阶段工作，不建议为了追求本批数据 100% 而提前引入。

因此，**值得修改候选 fuzzy 规则，但不值得通过删除负例、放宽 gold 或单独调低/调高阈值来追求 100%**。`中国农业银行`还应在后续标注评审中保留“严格词表 NIL 与现实简称覆盖”这一争议注记。
