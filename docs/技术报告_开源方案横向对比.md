## 2. 方案逐一分析

### 2.1 BLINK（Billion-scale Entity Linking）

**项目地址**: https://github.com/facebookresearch/BLINK

**架构**:

```
文本输入 → Bi-encoder (双塔BERT, 召回Top-K)
        → Cross-encoder (交互式BERT, 精排Top-1)
        → 链接结果
```

| 组件 | 实现 |
|------|------|
| 候选召回 | Bi-encoder: mention encoder + entity encoder, 点积相似度, FAISS ANN 索引 |
| 精排 | Cross-encoder: [CLS] mention [SEP] entity_desc [SEP] → Linear → sigmoid |
| 训练数据 | Wikipedia + Wikidata 自动生成 (distant supervision) |
| 实体规模 | 支持 6M 实体 |

**对课题10的适配评估**:

| 维度 | 评分 | 说明 |
|------|:---:|------|
| 架构适配性 | ★★★★★ | 两阶段架构是 EL 系统经典范式, 强烈建议借鉴 |
| 中文支持 | ★★☆ | 不支持中文, 需重新训练中文 Bi-encoder + Cross-encoder |
| 自定义 KB | ★★☆ | 面向 Wikipedia/Wikidata, KB 替换需大量改造 |
| 部署成本 | ★★★ | GPU 推理（Bi-encoder 可预计算实体向量缓存） |
| 消歧精度 | ★★★★★ | Wikipedia benchmark SOTA |
| NIL 检测 | ★★☆ | 不包含 NIL 模块, 默认假设所有 mention 可链接 |
| 推理速度 | ★★★★ | 实体向量预计算 + FAISS, 召回快 |
| 可解释性 | ★★ | 仅有相似度得分, 无可审计证据链 |

**借鉴要点**: Bi-encoder + Cross-encoder 两阶段架构设计；distant supervision 训练数据自动生成方法。

**不适用原因**: 无中文支持；无 NIL 检测；KB 改造工程量大。2021 年后维护停滞。

---

### 2.2 REL（Radboud Entity Linker）

**项目地址**: https://github.com/informagi/REL

**架构**:

```
文本 → Flair NER (mention detection)
     → 候选生成 (别名表 + Wikipedia API)
     → 实体消歧 (多种 embedding: Wikipedia2Vec, GloVe, BERT)
     → 链接结果
```

| 组件 | 实现 |
|------|------|
| NER | Flair NER |
| 候选召回 | 别名词典 + Wikipedia API 在线查询 |
| 消歧 | 支持 Wikipedia2Vec, GloVe, BERT 等多种 embedding |
| KB | Wikipedia 2019 dump (~100GB) |

**对课题10的适配评估**:

| 维度 | 评分 | 说明 |
|------|:---:|------|
| 架构适配性 | ★★★★ | 模块化设计好, 各部分可替换, NER+EL 一站式 |
| 中文支持 | ★★☆ | 弱, 主要面向英文 Wikipedia |
| 自定义 KB | ★★☆ | Wikipedia 强耦合, 替换 KB 需深入改造 |
| 部署成本 | ★★ | Wikipedia dump ~100GB, 首次加载慢 |
| 消歧精度 | ★★★★ | Wikipedia 场景表现好 |
| NIL 检测 | ★★★ | mention detection 含过滤, 但非专门的 NIL 模块 |
| 推理速度 | ★★★ | 中等, 首次加载慢 |
| 可解释性 | ★★★ | 提供 Python API 和 REST API |

**借鉴要点**: 模块化设计理念；NER+EL 集成方式；API 封装方式。

**不适用原因**: 强 Wikipedia 依赖；知识库庞大（~100GB）；中文支持弱。

---

### 2.3 GENRE（Generative Entity Retrieval）

**项目地址**: https://github.com/facebookresearch/GENRE

**架构**:

```
文本 + mention → BART Seq2Seq → 自回归生成实体名称
                                    ↓
                            Trie 约束解码 (保证输出在 KB 中)
```

**核心创新**: 将 EL 从"检索+排序"转变为"生成"范式。通过 Trie（前缀树）约束 beam search, 确保生成的每个 token 路径都对应知识库中存在的实体名称。

| 组件 | 实现 |
|------|------|
| 模型 | BART-base / BART-large |
| 解码 | Constrained Beam Search（Trie 约束） |
| 输入 | `[START] mention_text [SEP] context [END]` |
| 输出 | 实体名称字符串 |

**对课题10的适配评估**:

| 维度 | 评分 | 说明 |
|------|:---:|------|
| 架构适配性 | ★★★★ | 端到端, 无需显式候选生成和排序, 统一框架 |
| 中文支持 | ★★★ | 中等, 可中文化训练但需要中文 BART + 中文 Trie |
| 自定义 KB | ★★☆ | KB 更新需要重建 Trie, 增量更新困难 |
| 部署成本 | ★★★ | 单模型, 不需要向量索引, 内存效率高 |
| 消歧精度 | ★★★★★ | 多个 benchmark SOTA |
| NIL 检测 | ★★★ | 可通过 Trie 外 token 或阈值判断 |
| 推理速度 | ★★ | 自回归生成, 比 Bi-encoder 慢 |
| 可解释性 | ★★★ | 生成结果可直接阅读 |

**借鉴要点**: 生成式 EL 思路；constrained decoding 方法；统一 mention detection + EL 框架。

**不适用原因**: 推理慢（自回归）；KB 更新需重建 Trie；工业落地延迟是瓶颈。

### 2.4 OpenTapioca

**项目地址**: https://github.com/wetneb/opentapioca

**架构**:

```
mention → Solr 全文检索 (实体名称+别名索引)
       → 简单特征工程 (名称相似度 + 先验概率)
       → 链接结果
```

**对课题10的适配评估**:

| 维度 | 评分 | 说明 |
|------|:---:|------|
| 架构适配性 | ★★★ | 基于搜索引擎的实用方案, 简单直接 |
| 中文支持 | ★★☆ | Solr 中文分词需自行配置 |
| 自定义 KB | ★★★ | 基于 Solr 索引, 替换 KB 相对容易 |
| 部署成本 | ★★★★ | 仅需 Solr, 轻量 |
| 消歧精度 | ★★★ | 低于深度学习方法 |
| NIL 检测 | ★★ | 无专门 NIL 模块 |
| 推理速度 | ★★★★ | Solr 检索快 |
| 可解释性 | ★★★ | 基于搜索得分, 可解释 |

**借鉴要点**: 基于搜索引擎做 EL 的思路（ES/Solr → BM25 + 别名索引）。

**不适用原因**: 精确度低；维护不活跃；方案较老旧。

### 2.5 spaCy Entity Linking

**项目地址**: https://spacy.io/

**架构**:

```
spaCy NLP Pipeline → NER → Entity Linking 组件
                             ├── 别名表先验概率 (Wikipedia anchor text)
                             └── 上下文相似度 (cosine)
```

**对课题10的适配评估**:

| 维度 | 评分 | 说明 |
|------|:---:|------|
| 架构适配性 | ★★★ | Pipeline 组件化, 与 spaCy 生态集成好 |
| 中文支持 | ★ | 极弱, 主要面向英文 Wikipedia |
| 自定义 KB | ★ | KB 固定, 几乎不可替换 |
| 部署成本 | ★★★★ | spaCy 安装即可用 |
| 消歧精度 | ★★★ | 中等, Wikipedia 场景 |
| NIL 检测 | ★★ | 阈值过滤 |
| 推理速度 | ★★★★ | 快 |
| 可解释性 | ★★★ | API 简单（一行代码） |

**借鉴要点**: API 设计极简。

**不适用原因**: 不支持自定义 KB；中文支持极弱。

### 2.6 Haystack

**项目地址**: https://github.com/deepset-ai/haystack

**架构**:

```
Document Store (Elasticsearch/OpenSearch)
    ↓
Retriever (BM25 / Dense / Hybrid)
    ↓
Reader/Ranker (Cross-encoder / LLM)
    ↓
Pipeline (组件自由组合)
```

**对课题10的适配评估**:

| 维度 | 评分 | 说明 |
|------|:---:|------|
| 架构适配性 | ★★★★ | Pipeline 设计理念优秀, 组件可自由组合 |
| 中文支持 | ★★★ | 中等, ES 中文分词 + 多语言 embedding |
| 自定义 KB | ★★★★ | 通过 Document Store 写入自定义实体 |
| 部署成本 | ★★★ | ES + 向量库 部署成本中等 |
| 消歧精度 | ★★★ | EL 非核心功能, 精度依赖组件选择 |
| NIL 检测 | ★★ | 不直接支持 |
| 推理速度 | ★★★★ | ES 成熟稳定 |
| 可解释性 | ★★★ | 可记录检索路径 |

**借鉴要点**: Pipeline 组件化框架设计；多后端可插拔；混合检索方法。

**不适用原因**: EL 不是核心功能, 相关组件较基础；缺乏专门的 EL 模型集成。

### 2.7 LangChain 驱动的 EL

**典型实现**:

```python
# 候选召回
candidates = vectorstore.similarity_search(mention, k=10)
# LLM 消歧
result = llm.invoke(f"""
Given the mention "{mention}" in context "{context}",
which entity does it refer to?
Candidates: {candidates}
Return the entity_id and your reasoning.
""")
```

**对课题10的适配评估**:

| 维度 | 评分 | 说明 |
|------|:---:|------|
| 架构适配性 | ★★★ | 快速原型开发, 但不适合生产 |
| 中文支持 | ★★★★ | LLM 中文能力强（GPT-4/Claude/Qwen） |
| 自定义 KB | ★★★★ | 向量数据库可自由写入 |
| 部署成本 | ★★★★ | 向量 DB 简单, LLM API 成本高 |
| 消歧精度 | ★★★★★ | LLM 零样本/few-shot 可达或超专门模型 |
| NIL 检测 | ★★★★ | LLM 可判断 KB 中是否存在 |
| 推理速度 | ★ | 秒级, 不适合实时 |
| 可解释性 | ★★★★★ | LLM 可给出详细推理过程 |

**借鉴要点**: LLM 作为疑难 cases 兜底判断；few-shot prompt 设计。

**不适用原因**: 延迟高、成本高；输出不稳定；不适合作为主力链路。

## 4. 选型结论与推荐架构

### 4.1 核心结论

> **不建议照搬任何一个开源方案, 但可以从多个方案中借鉴关键设计。**

原因：
1. **所有方案均未同时满足"中文 + 自定义KB + NIL检测 + 可审计"** 四个课题10的硬约束；
2. 学术界方案（BLINK/GENRE/REL）面向 Wikipedia, 与行业 KB 场景差异大；
3. 工业界轻量方案（OpenTapioca/spaCy）精度不足, 中文支持弱；
4. LLM 方案（LangChain）延迟/成本不适合做主力链路。

### 4.2 推荐自研架构（借鉴组合）

```
┌──────────────────────────────────────────────────┐
│                  课题10 实体链接架构                │
│                                                    │
│  借鉴 BLINK: Bi-encoder + Cross-encoder 两阶段      │
│  借鉴 Haystack: Pipeline 组件化 + 多后端可插拔       │
│  借鉴 GENRE: Trie 约束解码 → 用于标准化别名扩展      │
│  借鉴 LangChain: LLM 作为疑难 cases 兜底            │
│                                                    │
│  Layer 1 (80%): 别名精确匹配 → 借鉴 ES BM25         │
│  Layer 2 (15%): Bi-encoder 语义召回                 │
│  Layer 3 (4%):  Cross-encoder 精排                  │
│  Layer 4 (1%):  LLM 兜底消歧                        │
│                                                    │
│  自研部分:                                          │
│  ├── 中文分词 + 别名模糊匹配 (行业特化)               │
│  ├── NIL 检测二分类器 (阈值 + top_score_ratio)       │
│  ├── 可审计证据链 (alias_match + context_match)     │
│  ├── 知识库热插拔 (JSONL schema + 行业切换)          │
│  └── SQLite 全链路留痕 (治理场景刚需)                 │
└──────────────────────────────────────────────────┘
```

### 4.3 关键技术决策

| 决策点 | 选择 | 理由 |
|------|------|------|
| 候选召回主力 | Elasticsearch BM25 + 别名词典 | 中文友好, 部署成熟, 可解释 |
| 语义召回增强 | BGE-M3 (BAAI) | 开源多语言, 中文表现好, 1024d |
| 精排方案 | Cross-encoder (轻量 BERT) | 候选 < 20 时可接受 |
| LLM 兜底 | Qwen / 本地部署 | 数据不出域, 成本可控 |
| 向量索引 | FAISS (CPU) | 103 实体规模, 无需 GPU |
| 知识库格式 | JSON Lines + 内存索引 | 热插拔, 版本化, 易分发 |
| NIL 判定 | 阈值法 (0.90) + top_score_ratio | 当前规模够用, 后续升级二分类器 |

### 4.4 从 MVP 到生产级的演进路径

```
MVP (当前阶段):
  别名词典 + ES BM25 + fallback 规则消歧 + SQLite 留痕
  实体规模: 103 → 1K

V1 (下一迭代):
  + BGE-M3 语义召回 + Cross-encoder 精排 + LLM 兜底
  实体规模: 1K → 100K

V2 (生产级):
  + 多源知识融合 (跨行业 KB 对齐)
  + 图神经网络集体消歧
  + 实体规模: 100K → 10M
```
