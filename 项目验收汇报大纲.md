# EL-KA 本地化实体链接系统阶段验收汇报大纲

> 这份大纲针对“本地可私有化部署的实体链接 HTTP 服务”场景重构，强调项目的核心能力、现场演示路径与验收证据，而不是泛化地介绍开发分工。

---

## 一、项目背景与验收目标（2分钟）

1. 项目背景
- 面向企业/研究场景，构建一个可本地部署、可私有化使用的实体链接系统。
- 目标是实现“文本 → mention → 候选实体 → 消歧结果 → NIL 判定 → 共指补全 → 可追溯输出”的完整链路。

2. 验收目标
- 支持中文实体识别与归一化链接。
- 支持别名匹配、候选生成、实体消歧与 NIL 判定。
- 支持共指消解（规则共指）与全链路 trace 日志。
- 提供 FastAPI 接口，便于本地调试、业务集成与后续扩展。

3. 项目价值
- 解决“实体名称不统一、别名多样、同义指代难以判断”的问题。
- 将实体链接能力封装为标准 HTTP 服务，便于接入业务系统或 Agent 工作流。

---

## 二、系统整体架构与技术路线（3分钟）

1. 整体架构
- 知识库层：提供行业实体数据与别名索引。
- 实体匹配层：支持别名精确/模糊匹配、向量检索与消歧排序。
- 共指模块：处理“他/他们/这些机构”等指代关系。
- 服务层：基于 FastAPI 封装统一接口。
- 日志与追溯层：通过 SQLite 记录请求、实体、结果与中间步骤。
- Agent 层：提供可扩展的智能体/调度入口，便于后续接入不同推理策略。

2. 核心技术路线
- 本地规则回退链路：别名匹配 + 规则消歧 + NIL 判定。
- 模型链路（可选）：BGE 向量检索 + 重排序 + LLM/规则兜底。
- 共指链路：基于已链接先行词进行规则回指。

3. 交付形态
- 可运行的 Python 服务。
- 可直接复现的样例请求与测试脚本。
- 可审计的 trace 数据库与接口文档。

---

## 三、项目核心能力展示（8分钟）

### 1. 实体链接主流程
- 输入为文本 + mentions + options。
- 经过候选生成、消歧、NIL 判定后输出标准实体 ID 与标准名。
- 结果支持返回 evidence、候选列表与 link_basis，便于人工复核。

### 2. 共指消解能力
- 重点演示“前文实体 + 代词/集合指代”的场景。
- 例如：
  - “人民日报社、新华通讯社、上海报业集团澎湃新闻正在飞速发展，他们是我们的未来。”
- 展示共指开关开启/关闭后的区别，说明为什么共指必需建立在前面实体已成功链接的基础上。

### 3. 可追溯能力
- 每一次请求都会生成 trace_id。
- 结果、候选、链路步骤可在 SQLite 数据库中回放。
- 这使得系统不仅“能返回结果”，还“能解释结果如何生成”。

### 4. 服务化能力
- 通过 HTTP 接口直接调用。
- 可用于前端调试、Postman 测试、业务代码集成。
- 接口规范统一，Swagger 可视化，便于交付与二次开发。

---

## 四、模块成果与重点实现（按项目真实结构来讲）

### 模块一：实体链接流水线
- 实现主流程串联：候选生成 → 消歧 → NIL 判定。
- 支持本地 fallback 规则链路，适合演示与可控环境验证。
- 支持 EntityAlignmentV0 组件接入，具备模型/向量链路扩展能力。

### 模块二：知识库与别名处理
- 支持外部知识库路径动态传入。
- 知识库中包含实体、别名与类型信息。
- 适配 JSON 结构，便于行业数据替换与扩展。

### 模块三：共指模块
- 针对中文代词、名词性指代、集合指代做规则性回指。
- 对“他们 / 这些机构”等场景进行规则化处理。
- 通过开关控制是否启用，兼容不同演示场景与测试需求。

### 模块四：FastAPI 服务与数据库
- 提供健康检查、实体链接主接口与 Agent 查询入口。
- 自动初始化 SQLite trace 数据库。
- 支持请求级别的 trace_id 追踪。

### 模块五：Agent 调度框架
- 实现 Agent 工厂注册与可插拔策略入口。
- 支持默认本地引擎与后续扩展自定义推理策略。
- 便于把本项目作为 Agent/业务系统的底层实体能力组件。

## 五、现场演示流程建议（最关键）

### 演示 1：基础实体链接
- 输入一段简短文本，展示实体链接结果。
- 重点看：entity_id、standard_entity、confidence、is_nil。

### 演示 2：共指开关对比
- 同一段文本分别在共指关闭/开启下展示结果差异。
- 重点展示共指能够为代词/集合指代补出更合理结果。

### 演示 3：NIL 识别效果
- 通过低置信或无候选场景，展示系统如何返回 NIL。
- 重点说明 NIL 不是失败，而是系统对“不可链接”进行明确判定。

### 演示 4：Trace 回放
- 展示 trace_id 对应的请求记录与链路信息。
- 说明系统不只“输出结果”，还提供“结果可解释性”。

### 演示 5：服务接口调用
- 使用 curl / Postman / Swagger 展示接口可调用性。
- 说明项目已经具备可直接交付的服务化能力。

## 六、测试与验收证据（2分钟）

1. 自动化测试覆盖范围
- 服务健康检查。
- 实体链接主流程。
- 共指基础场景。
- 知识库加载与配置校验。
- API 接口与 Agent 注册逻辑。

2. 现有可用证据
- 评测脚本与报表文件。
- 规则共指相关评测结果。
- trace 数据库中的实际请求样例。
- 运行中的服务接口调用结果。

3. 验收说明
- 重点在于“链路完整、可演示、可追溯、可扩展”，而非单纯追求某一模块极致精度。
- 由于本项目同时兼顾本地低成本部署与模型扩展能力，因此验收更适合以“功能完备性 + 演示可运行性 + 追溯能力”为核心。

## 八、答疑与汇报建议

1. 汇报重点建议
- 不要把汇报写成“纯理论介绍”，而要突出“系统如何从输入文本走到最终实体结果”。
- 重点讲清楚：项目解决了什么问题、为什么它能落地、为什么它能被演示出来。

2. 适合现场重点强调的三句话
- “这是一个可私有化部署的中文实体链接服务。”
- “它不仅能返回实体链接结果，还能追溯结果如何生成。”
- “它已经具备从规则链路到后续模型扩展的基础能力。”

---

## 九、建议的汇报顺序（可直接照着讲）

1. 先讲项目目标与痛点。
2. 再讲系统架构与核心能力。

4. 接着展示接口调用与 trace 回放。
5. 最后补充测试与交付成果。
